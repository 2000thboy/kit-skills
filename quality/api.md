# kit-skills v2.0

# API 一致性检查清单

确保前后端 API 契约一致、错误处理完整、认证正确、性能策略到位。

---

## 1. API 路径与架构文档一致

**检测方法**
- [ ] 对比 `docs/architecture/api.md` 中的路径定义和代码中的实际调用
- [ ] 检查 RESTful 规范：资源名词复数、HTTP 方法语义正确
- [ ] 检查版本控制（如 `/v1/users` vs `/users`）
- [ ] 检查是否有未文档化的 "幽灵接口"

**常见错误模式**
- 文档写 `GET /api/users`，代码写 `GET /api/user/list`
- 混合 REST 和 RPC 风格（如 `POST /api/users/createUser`）
- 版本号不一致（部分接口带 `/v1/`，部分不带）
- 前端调用未文档化的内部接口

**修复建议**
- 建立 API 路径规范文档，前后端共同遵守
- 使用 OpenAPI / Swagger 定义契约，生成前后端类型
- 路径统一使用复数名词：`/users`、`/orders`、`/products`
- HTTP 方法语义：GET（读）、POST（创建）、PUT/PATCH（更新）、DELETE（删除）

**严重程度**: P0（路径不一致 = 调用失败）

---

## 2. 请求/响应格式符合契约

**检测方法**
- [ ] 对比 API 文档中的请求体定义和实际发送的 payload
- [ ] 对比 API 文档中的响应体定义和实际接收的数据
- [ ] 检查 Content-Type（`application/json`）
- [ ] 检查字符编码（UTF-8）

**常见错误模式**
- 前端发送 `camelCase`，后端期望 `snake_case`
- 前端发送嵌套对象，后端期望扁平结构
- 响应缺少前端期望的字段（如 `created_at`、`id`）
- 数组字段有时返回 `[]`，有时返回 `null`
- 数字字段后端返回 string（如 `"price": "99.99"`）

**修复建议**
- 使用 Zod / Yup / io-ts 在 API 层做请求/响应校验
- 建立自动转换层（camelCase ↔ snake_case）
- 响应字段缺失时提供默认值，而非直接报错
- 统一空数组返回 `[]`，不返回 `null`
- 使用 `openapi-typescript` 从契约生成类型

**严重程度**: P0（格式不一致 = 运行时错误）

---

## 3. 错误码处理完整

**检测方法**
- [ ] 检查代码中是否处理了 400、401、403、404、500
- [ ] 模拟每种错误码，检查前端行为
- [ ] 检查是否有未处理的错误码（如 409 Conflict、422 Unprocessable Entity）
- [ ] 检查错误处理是否覆盖所有 API 调用路径

**常见错误模式**
- 只处理 200 和 500，忽略 400/401/403/404
- 401 未触发重新登录，页面白屏
- 403 显示 "无权限" 但不阻止用户继续操作
- 404 显示 "页面不存在" 但 URL 是对的（实际是 API 404）
- 网络错误（无状态码）未被 catch

**修复建议**

| 状态码 | 场景 | 前端处理 |
|--------|------|----------|
| 400 | 参数验证失败 | 显示字段级错误信息 |
| 401 | 未认证/Token 过期 | 跳转登录或显示登录弹窗 |
| 403 | 无权限 | 显示 "无权限" + 联系管理员 |
| 404 | 资源不存在 | 显示 "资源不存在" |
| 409 | 资源冲突（如重复创建） | 显示具体冲突原因 |
| 422 | 语义错误（如业务规则 violation） | 显示业务错误信息 |
| 429 | 请求过多 | 显示 "操作太频繁，请稍后" |
| 500 | 服务端错误 | 显示 "服务暂时不可用" + 错误 ID |
| 503 | 服务维护 | 显示 "服务维护中" |
| 网络错误 | 无网络/超时 | 显示 "网络连接失败，请重试" |

**严重程度**: P0（错误未处理 = 用户困惑、安全漏洞）

## 4. 认证/授权正确传递

**检测方法**
- [ ] 检查请求头中是否有正确的认证信息
- [ ] 检查 Token 是否过期后自动刷新
- [ ] 检查未登录用户访问受保护路由是否被正确拦截
- [ ] 检查权限不足用户是否能看到/操作无权限的功能

**常见错误模式**
- Token 写在代码中（硬编码）
- Token 过期后不刷新，API 持续 401
- 认证信息通过 URL query 传递（不安全）
- 前端路由守卫和实际 API 权限不一致（前端隐藏了按钮但 API 未鉴权）
- 敏感操作（删除）无二次确认和无额外权限校验

**修复建议**
- Token 存储在 httpOnly cookie（推荐）或 localStorage
- 实现 Token 自动刷新机制（refresh token）
- 认证信息通过 `Authorization: Bearer <token>` 头部传递
- 前端路由守卫 + 后端 API 双重鉴权
- 敏感操作添加二次确认弹窗 + 后端再次校验权限

**严重程度**: P0（认证问题 = 安全漏洞）

---

## 5. 超时和重试策略

**检测方法**
- [ ] 检查 API 客户端是否配置了超时时间
- [ ] 模拟慢速网络，检查是否触发超时
- [ ] 模拟请求失败，检查是否自动重试
- [ ] 检查重试次数和间隔是否合理

**常见错误模式**
- 无超时配置，请求永远挂起
- 超时时间过短（如 3 秒），正常请求也超时
- 无限重试，导致服务器雪崩
- 重试间隔固定，未使用指数退避
- POST/PUT/DELETE 等副作用操作也重试（可能导致重复提交）

**修复建议**
- 配置合理的超时时间：GET 10-30s，POST/PUT 30-60s
- 实现指数退避重试：1s → 2s → 4s → 8s，最多 3 次
- 仅对 GET 等幂等操作自动重试
- 对 POST/PUT/DELETE 使用请求去重（idempotency key）
- 超时时显示友好提示 + 重试按钮

**严重程度**: P1（影响可用性和服务器稳定性）

---

## 6. 防抖/节流是否正确

**检测方法**
- [ ] 在搜索框快速输入，检查 API 请求次数
- [ ] 快速点击提交按钮，检查是否发送多次请求
- [ ] 滚动加载时，检查是否触发过多请求
- [ ] 窗口 resize 时，检查是否触发过多重计算

**常见错误模式**
- 搜索框每输入一个字符就发一次请求（无防抖）
- 用户快速双击提交按钮，创建两条重复记录
- 滚动事件未节流，触发大量请求
- 防抖/节流时间设置不合理（如 5 秒太长，100ms 太短）

**修复建议**
- 搜索输入：防抖 300-500ms（用户停止输入后触发）
- 提交按钮：点击后立即禁用（loading 状态），防止重复提交
- 滚动加载：节流 200-300ms 或使用 Intersection Observer
- 窗口 resize：防抖 200ms
- 使用 lodash debounce/throttle 或框架内置 hook（如 `useDebounce`）

**严重程度**: P1（影响性能和用户体验）

---

## 7. API 变更是否更新了 SPEC.md

**检测方法**
- [ ] 对比当前代码中的 API 调用和 `docs/architecture/api.md`
- [ ] 对比当前代码中的 API 调用和 `.plan/SPEC.md`
- [ ] 检查是否有新增/修改/删除的 API 未在文档中体现
- [ ] 检查 API 变更是否同步到前端类型定义

**常见错误模式**
- 代码中 API 已变更，但文档仍是旧版
- 新增接口未写入 SPEC
- 删除的接口仍在文档中
- 字段变更未同步到 TypeScript 类型

**修复建议**
- 任何 API 变更必须同步更新 `docs/architecture/api.md`
- 使用 OpenAPI / Swagger 作为单一事实源
- API 变更触发前端类型重新生成
- 在 PR 检查清单中添加 "API 文档已更新" 项

**严重程度**: P1（文档不同步 = 维护成本增加）

## 检查汇总表

| 检查项 | 检测方法 | 严重程度 | 常见错误 |
|--------|----------|----------|----------|
| API 路径一致 | 对比文档和代码 | P0 | 路径拼写不一致、风格混用 |
| 请求/响应格式 | Payload 对比 + 类型检查 | P0 | camelCase vs snake_case |
| 错误码完整 | 模拟各状态码 | P0 | 只处理 200/500 |
| 认证/授权 | 检查请求头 + 权限测试 | P0 | Token 硬编码、未鉴权 |
| 超时和重试 | 模拟慢网 + 失败 | P1 | 无超时、无限重试 |
| 防抖/节流 | 快速操作测试 | P1 | 无防抖、重复提交 |
| API 文档同步 | 文档 vs 代码对比 | P1 | 文档过时 |
