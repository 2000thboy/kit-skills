# OpenCLI Framework

## 一句话

OpenCLI 类框架适合做本地或浏览器自动化执行：打开页面、点按钮、读 DOM、抓网络请求、上传内容、提交表单、保存证据。

它不是产品规格框架，也不是质量审查框架。它负责动手，不负责判断这件事该不该做。

## 适合什么

- 浏览器操作和平台后台自动化。
- 发布、上传、提交、抓取、状态检查。
- 保存截图、网络日志、页面证据。
- 作为项目 runner 的一个执行节点。

## 高风险点

- 账号材料不能入库。
- dry-run 和 live 必须隔离。
- 平台限额、审核、风控、验证码、人工确认必须能停住。
- DOM 选择器和 SPA 行为容易变，不能把一次跑通当永久可靠。

## 和 KIT 的关系

KIT 记录：

- 哪个外发、提交、发布或浏览器节点使用 OpenCLI 或项目脚本。
- 是否 live。
- 人工确认门在哪里。
- 失败后如何恢复状态。
- 证据写到哪里。

OpenCLI 执行：

- 浏览器动作。
- 平台提交。
- 页面和网络证据采集。
- 必要时返回 `CONFIRM_REQUIRED`、`DEFERRED`、`BLOCKED`。

## 长流程内容 workflow 默认门禁

长流程内容 workflow 不一定要发布。它可能只是每天生成日报、每周生成报告、批量整理摘要、产出文件、提交到某个系统，或者最后才发布。只要涉及周期性生成、长链路、多步骤审查、外部写入或可恢复运行，就应该有这些门禁：

- 真实外发、写入、提交或发布确认门。
- dry-run/live 隔离。
- 账号材料不入库。
- 状态恢复。
- 平台额度和 `DEFERRED` 行为。
- 质量阻断。
- 运行证据、产物证据和必要的外发/发布证据。
