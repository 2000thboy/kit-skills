# 常见场景 Loop 配方

预定义的命令组合和执行路径，用于快速进入标准工作流。

---

## 配方 1：前端页面开发 loop

```
/kit-run start → 按 SPEC 实现页面 → /kit-check diff → 发现问题 → 修复 → /kit-check diff → 通过 → 沉淀
```

- **适用场景**: 新增或修改前端页面，需要确保 UI 质量和代码规范
- **前置条件**:
  - `.plan/SPEC.md` 已定义页面需求和验收标准
  - `docs/ui-ux/` 有设计参考或品牌色板
  - 开发环境已就绪（`npm run dev` 可启动）
- **执行步骤**:
  1. `/kit-run start`：读取 SPEC，按任务列表开始编码
  2. 编码前过 5 步门禁（`quality/pre-code.md`）
  3. 每写一个文件前过 4 项自检（`modes/run.md`）
  4. 前端优先：先实现 UI → 截图检查 → 用户确认 → 再联调
  5. `/kit-check diff`：对变更文件进行增量质量检查
  6. 发现问题 → 修复 → 再次 `/kit-check diff`
  7. 通过后运行实现闭环 5 项检查（`quality/post-code.md`）
  8. 沉淀：将新发现的模式写入 `.kit/quality-patterns.md`
- **退出条件**:
  - `/kit-check diff` 无 P0/P1 问题
  - build 无错误，lint 无 error
  - 截图验证通过（mobile + desktop）
  - 用户确认 UI 符合预期
- **常见陷阱**:
  - 跳过移动端检查：只在 desktop 测试，上线后移动端崩坏
  - 忽略空状态：有数据时正常，无数据时白屏
  - mock 数据残留："先 mock 再联调"变成永久 mock
  - 硬编码颜色：直接写 `#3B82F6` 而不是设计 token

## 配方 2：API 联调 loop

```
/kit-run start → 实现 API 调用 → /kit-check full（数据层重点）→ 发现 mock 残留 → 替换为真实 API → 回归 → 通过
```

- **适用场景**: 前后端接口对接，替换 mock 数据为真实 API
- **前置条件**:
  - 后端 API 文档就绪（Swagger/OpenAPI 或接口文档）
  - `.plan/SPEC.md` 已定义数据模型和 API 契约
  - 前端已有 mock 实现或静态数据
- **执行步骤**:
  1. `/kit-run start`：读取 SPEC，定位需要联调的 API 点
  2. 实现真实 API 调用，配置 `API_BASE_URL`
  3. `/kit-check full`：重点检查数据层（`quality/data.md`）
  4. 检查清单：
     - 所有 CRUD 操作调用真实后端？
     - 表单提交后数据真的保存了吗？
     - 刷新页面数据还在吗？
     - 错误状态用户可见吗？
     - 有 loading 状态吗？
  5. 发现 mock 残留 → 替换为真实 API
  6. 回归测试：所有相关功能重新验证
  7. 通过后更新 `.plan/CHECKLIST.md` 联调状态
- **退出条件**:
  - 所有 API 调用指向真实后端
  - 无 mock 数据残留在生产代码中
  - 错误处理完整（401/403/500 都有用户可见提示）
  - 请求有超时和重试机制
- **常见陷阱**:
  - 部分联调：只联调了主要接口，边缘接口仍用 mock
  - 忽略错误处理：只测试 happy path，不测试失败场景
  - 类型不同步：前端类型定义和后端实际返回不一致
  - 环境配置遗漏：API 基础路径硬编码，切换环境困难

## 配方 3：UI 问题修复 loop

```
用户报告"按钮位置不对" → /kit-check diff（触发发散检查）→ 检查所有按钮/表单/移动端/z-index → 产出报告 → 条件确认 → 修复 → 回归 → 沉淀到 .kit/quality-patterns.md
```

- **适用场景**: 用户报告具体 UI 问题，需要系统性排查同类问题
- **前置条件**:
  - 用户提供了具体问题描述或截图
  - 可访问相关页面进行验证
- **执行步骤**:
  1. 接收用户报告，记录到 `.kit/quality-patterns.md`
  2. `/kit-check diff`：触发发散检查（divergent inspection）
  3. 发散检查范围：
     - 所有按钮的对齐和间距
     - 所有表单的布局和验证提示
     - 移动端视图下的相同元素
     - z-index 层级冲突
     - 文本溢出和截断
     - 响应式断点异常
  4. 产出质量报告（`templates/quality-report.md`）
  5. 用户确认修复范围（全部修复或只修复报告的部分）
  6. 修复问题
  7. `/kit-check` 回归检查
  8. 通过后沉淀：将 bug 模式写入 `.kit/quality-patterns.md`
- **退出条件**:
  - 用户确认的问题已修复
  - 发散检查发现的同类问题已处理或明确标记 deferred
  - 截图验证通过
- **常见陷阱**:
  - 只修用户指出的一个点，忽略同类问题
  - 修复引入新问题（回归）
  - 不发散检查，导致用户反复报告类似问题
  - 修复后不做截图验证

## 配方 4：自动巡航 loop

```
/kit-loop 1day → 用户确认 → 启动 → [每4小时检查点] → 自动 /kit-check diff → 修复 → 继续 → [用户可随时 stop] → 最终报告
```

- **适用场景**: 长时间运行的开发任务，需要自动迭代和自我修复
- **前置条件**:
  - 任务目标明确，已建档（PRD/SPEC/CHECKLIST）
  - 用户授权自动修复权限（范围限定）
  - 有明确的终止条件（时间、里程碑、问题清零）
- **执行步骤**:
  1. `/kit-loop 1day`：声明巡航时长和任务范围
  2. 用户确认：授权范围、检查点频率、通知方式
  3. 启动巡航：按 SPEC 任务列表自动执行
  4. 每 4 小时检查点：
     - 自动运行 `/kit-check diff`
     - 产出检查报告
     - 自动修复 P0 问题（在授权范围内）
     - 记录修复日志到 `.kit/loop-state.md`
  5. 发现问题 → 自动修复 → 继续执行
  6. 用户可随时 `stop` 中断巡航
  7. 到达终止条件或用户 stop → 产出最终报告
- **退出条件**:
  - 到达预设时间上限
  - 完成所有 SPEC 任务
  - 用户主动 stop
  - 连续 2 轮检查无新问题（收敛退出）
- **常见陷阱**:
  - 授权范围不清：自动修复了不该修的文件
  - 检查点间隔过长：问题累积难以修复
  - 无明确终止条件：无限循环
  - 不记录日志：无法追溯自动修复的内容

## 配方 5：新项目启动 loop

```
/kit init → /kit brainstorm → /kit-run start（前端优先）→ /kit-check full → /kit archive
```

- **适用场景**: 从零开始的新项目，需要完整的产品到开发流程
- **前置条件**:
  - 用户有初步的产品想法
  - 已确定目标用户和核心痛点
  - 有开发环境（Node.js、代码编辑器等）
- **执行步骤**:
  1. `/kit init my-project`：初始化项目结构
     - 生成 `.plan/PRD.md`、`SPEC.md`、`CHECKLIST.md`
     - 生成 `.kit/config.json`、`version.json`
     - 生成 `.workflow/` 和 `.test/`
  2. `/kit brainstorm "我的想法"`：产品脑暴
     - 追问关键产品问题
     - 确定目标用户、核心痛点、第一条 workflow
     - 评估风险和技术路线
  3. `/kit-run start`：开始执行
     - 编码前 5 步门禁
     - 前端优先：先实现核心页面 UI
     - 截图检查 → 用户确认
     - 再实现后端和联调
  4. `/kit-check full`：完整质量检查
     - UI 层：遮挡、响应式、文本溢出、空状态
     - 数据层：mock 残留、表单反馈、分页、loading
     - API 层：路径配置、401/403、超时、防抖
     - 架构层：use client、import 路径、类型同步、死代码
  5. `/kit archive`：归档沉淀
     - 整理历史计划和证据
     - 更新 `.kit/quality-patterns.md`
     - 归档旧版本到 `.plan/archive/`
- **退出条件**:
  - `/kit-check full` 无 P0 问题
  - 用户确认核心功能可用
  - 所有文档（PRD/SPEC/CHECKLIST）与代码事实一致
- **常见陷阱**:
  - 跳过脑暴直接编码：需求不清导致返工
  - 不定义验收标准：无法判断"完成"
  - 忽略测试结构：`.test/` 未初始化，后期难以补
  - 不归档：历史计划和证据散落，无法追溯

## 配方选择速查

| 你的情况 | 使用配方 |
|---------|---------|
| 要新增/修改页面 | 配方 1：前端页面开发 loop |
| 要替换 mock 为真实 API | 配方 2：API 联调 loop |
| 用户报了 UI bug | 配方 3：UI 问题修复 loop |
| 要长时间自动开发 | 配方 4：自动巡航 loop |
| 从零开始新项目 | 配方 5：新项目启动 loop |
