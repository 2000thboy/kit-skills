# Vibe Coding 反模式库

记录 AI 编码中反复出现的问题模式。每个反模式包含：症状、危害、检测方法、修复方案、预防方法。

---

## UI 反模式

### 1. Emoji 图标代替专业图标库

- **症状**: 界面中出现 `🎉` `👍` `🔥` `✅` `❌` 等 emoji 作为功能图标或状态指示
- **危害**:
  - 不同操作系统/浏览器渲染差异大，Windows 和 macOS 的 emoji 风格完全不同
  - 可访问性差，屏幕阅读器无法正确描述
  - 无法调整颜色、大小、描边，与设计系统脱节
  - 显得不专业，降低产品可信度
- **检测方法**:
  - 全局搜索 Unicode emoji 字符范围 `\u{1F300}-\u{1F9FF}`
  - 搜索常见 emoji：`🎉👍🔥✅❌⚠️💡🚀⭐📌`
  - 检查 `package.json` 是否缺少 `lucide-react`、`@heroicons/react`、`@tabler/icons-react`
- **修复方案**:
  - 替换为 Lucide、Heroicons 或 Tabler Icons 的对应图标
  - 建立图标映射表（emoji → 图标库组件）
  - 统一使用 `<Icon name="..." />` 或 `<SomeIcon />` 模式
- **预防方法**:
  - 在项目初始化时锁定图标库（见 `modes/run.md` 图标库锁定）
  - 在 PreToolUse hook 中拦截 emoji 写入
  - 代码审查时检查新增文件是否包含 emoji

### 2. 紫色/粉色渐变 AI 模板

- **症状**: 大面积使用 `bg-gradient-to-r from-purple-500 to-pink-500` 或类似渐变
- **危害**:
  - 千篇一律的"AI 味"，用户一眼看出是模板
  - 与品牌色无关，无法建立品牌认知
  - 高饱和度渐变在暗色模式下刺眼
  - 打印/截图时颜色失真
- **检测方法**:
  - 搜索 `purple`、`pink`、`gradient` 在 CSS/JSX 中的组合使用
  - 检查 `tailwind.config.js` 是否定义了品牌色
  - 截图对比：如果页面看起来像"某个 AI 生成的模板"，就是问题
- **修复方案**:
  - 从品牌指南或设计 token 中提取主色/辅色
  - 使用单色或微渐变（同色系深浅变化）
  - 渐变只用于强调元素（按钮 hover、小标签），不用于背景
- **预防方法**:
  - 在 `docs/ui-ux/` 中定义品牌色板
  - 禁用 Tailwind 默认渐变工具类，只暴露设计 token
  - 设计审查时检查颜色使用

### 3. Card Wall 无信息层次

- **症状**: 所有卡片大小、颜色、边框完全相同，用户无法快速识别重点
- **危害**:
  - 信息密度高但可读性差
  - 用户无法区分"重要"和"次要"内容
  - 视觉疲劳，降低留存率
- **检测方法**:
  - 检查页面是否有超过 6 个完全相同的卡片
  - 检查是否有"视觉锚点"（突出的标题、不同的背景色、图标差异）
  - 让用户测试：能否在 3 秒内找到最重要的卡片？
- **修复方案**:
  - 引入信息层次：大卡片放重点内容，小卡片放次要内容
  - 使用颜色区分状态（成功/警告/错误）
  - 使用图标+文字组合，而非纯文字
  - 添加"推荐"、"热门"等标签
- **预防方法**:
  - 在 PRD 中定义信息优先级
  - 使用设计系统的 `size` token（sm/md/lg/xl）
  - 参考 Material Design 的信息层次规范

### 4. 硬编码颜色

- **症状**: 代码中直接写 `#3B82F6`、`text-blue-500`、`rgb(59,130,246)`
- **危害**:
  - 品牌色变更时需要全局替换，容易遗漏
  - 暗色模式适配困难
  - 不同组件颜色不一致
- **检测方法**:
  - 搜索 `#` 开头的 hex 颜色值（排除注释和 SVG）
  - 搜索 Tailwind 颜色类（`text-`/`bg-`/`border-` + 颜色名 + 数字）
  - 检查 `tailwind.config.js` 或 CSS 变量定义
- **修复方案**:
  - 提取为 CSS 变量或 Tailwind 自定义颜色
  - 使用语义化命名：`--color-primary`、`--color-surface`、`--color-text-primary`
  - 建立设计 token 文件
- **预防方法**:
  - 在 `modes/run.md` 中强制设计 token 检查
  - ESLint/Stylelint 规则禁止硬编码颜色
  - 代码审查时检查新增颜色使用

### 5. 响应式断点缺失

- **症状**: 只在 desktop 宽度开发和测试，移动端布局崩坏
- **危害**:
  - 移动端用户流失（通常占 50% 以上流量）
  - 横向滚动、文字过小、按钮点不到
  - 搜索引擎降权（移动优先索引）
- **检测方法**:
  - 在 320px、375px、768px、1024px、1440px 下检查页面
  - 检查是否有 `@media` 查询或 Tailwind 响应式前缀
  - 使用浏览器 DevTools 的设备模拟器
- **修复方案**:
  - 从 mobile-first 开始设计
  - 使用 `min-w-0`、`overflow-hidden` 防止 flex 子元素溢出
  - 表格在移动端转为卡片或横向滚动
  - 导航在移动端转为汉堡菜单或底部 tab
- **预防方法**:
  - 在 `modes/run.md` 前端优先流程中强制移动端检查
  - 截图验证必须包含移动端视图
  - 使用 Storybook 或类似工具测试多断点

### 6. z-index 战争

- **症状**: 代码中出现 `z-10`、`z-50`、`z-[999]`、`z-[9999]`、`z-[99999]`
- **危害**:
  - 层级关系不可维护，后来者只能加更大的数字
  - 弹窗被其他弹窗覆盖，模态框失效
  - 调试困难，无法预测最终层级
- **检测方法**:
  - 全局搜索 `z-index`、`z-`、`z-[`
  - 检查是否有层级管理系统
  - 检查 `z-[999` 或更大的魔法数字
- **修复方案**:
  - 定义层级系统：`base(0)`、`dropdown(100)`、`sticky(200)`、`modal(300)`、`popover(400)`、`tooltip(500)`
  - 使用 CSS 变量管理：`--z-modal: 300`
  - 弹窗使用 Portal 渲染到 body，避免嵌套 z-index
- **预防方法**:
  - 在 design token 中定义 z-index 层级
  - 禁止直接使用数字 z-index，只允许使用预定义变量
  - 使用 `headlessui` 或 `radix-ui` 等已处理层级的组件库

### 7. 文本溢出不处理

- **症状**: 长文本直接截断或撑破容器，依赖容器宽度
- **危害**:
  - 布局崩坏，卡片被撑大
  - 用户看到截断的句子，信息不完整
  - 表格列宽失控
- **检测方法**:
  - 用超长文本（100+ 字符）测试所有文本区域
  - 检查是否有 `overflow`、`text-overflow`、`line-clamp` 使用
  - 检查弹性布局中是否有 `min-width: 0`
- **修复方案**:
  - 单行截断：`truncate`（`overflow-hidden text-ellipsis whitespace-nowrap`）
  - 多行截断：`line-clamp-2`、`line-clamp-3`
  - 提供 tooltip 或展开按钮显示完整内容
  - 表格设置 `max-width` 和 `overflow-x-auto`
- **预防方法**:
  - 在组件库中封装 `Text` 组件，内置截断处理
  - 测试数据包含边界长度文本
  - 截图验证时检查文本区域

### 8. 空状态 = 白屏

- **症状**: 无数据时页面空白，用户不知道是没有数据还是加载失败
- **危害**:
  - 用户困惑，可能反复刷新
  - 看起来像 bug
  - 新用户不知道下一步该做什么
- **检测方法**:
  - 清空数据库/接口返回空数组，检查页面表现
  - 检查每个列表、表格、卡片区域是否有空状态处理
  - 检查首次使用引导
- **修复方案**:
  - 空状态组件：图标 + 说明文字 + 操作按钮
  - 示例："暂无订单" + "去购物" 按钮
  - 首次使用：引导用户创建第一个内容
  - 加载中 vs 空数据 vs 错误，三种状态区分清楚
- **预防方法**:
  - 在 PRD 中定义每个页面的空状态
  - 封装 `DataContainer` 组件，统一处理 loading/empty/error
  - 测试用例必须覆盖空状态

## 数据反模式

### 9. Mock 数据长期残留

- **症状**: `const mockData = [...]` 在代码中存在超过一个迭代周期，"先 mock 再联调"变成永久 mock
- **危害**:
  - 上线后用户看到假数据
  - 前后端联调时发现数据格式不一致
  - 无法测试真实边界情况
- **检测方法**:
  - 搜索 `mock`、`fake`、`dummy`、`sample` 在源码中的使用
  - 检查 API 调用是否被注释掉，旁边是 mock 数据
  - 检查 `.env` 或配置文件中是否有 `USE_MOCK=true`
- **修复方案**:
  - 建立联调检查点：每个 API 必须有真实调用证据
  - mock 数据移到 `.test/ai/fixtures/`，只在测试环境使用
  - 使用 MSW (Mock Service Worker) 统一管理 mock，明确标记
- **预防方法**:
  - 在 `modes/run.md` 实现闭环中检查"新增代码接入真实调用链"
  - CHECKLIST 中设置"mock 清理"任务项
  - 代码审查时标记 mock 数据

### 10. 表单提交后无反馈

- **症状**: 点击提交按钮后，页面没有任何变化，用户不知道成功还是失败
- **危害**:
  - 用户重复提交
  - 用户以为失败，放弃操作
  - 数据重复创建
- **检测方法**:
  - 测试所有表单：提交后是否有成功/失败提示
  - 检查是否有 `toast`、`notification`、`alert` 组件使用
  - 检查网络面板：请求成功后 UI 是否更新
- **修复方案**:
  - 成功：toast 提示 + 页面跳转/数据刷新
  - 失败：错误提示 + 保留表单数据 + 高亮错误字段
  - 提交中：按钮 loading 状态，禁用重复提交
- **预防方法**:
  - 封装 `Form` 组件，内置提交状态管理
  - 在 `modes/run.md` 写文件前自检中检查表单反馈
  - 每个表单在 PRD 中定义提交后的用户旅程

### 11. 前端假分页

- **症状**: 全部数据加载到前端后，用 JS 做分页显示
- **危害**:
  - 数据量大时页面卡顿，内存溢出
  - 初始加载时间过长
  - 无法利用数据库索引优化
- **检测方法**:
  - 检查 API 请求是否包含 `page`、`limit`、`offset` 参数
  - 检查前端分页组件是否接收全量数据
  - 检查网络面板：切换页码时是否有新请求
- **修复方案**:
  - 后端实现真分页：SQL `LIMIT/OFFSET` 或 cursor 分页
  - 前端分页组件只接收当前页数据
  - 大数据量时使用虚拟滚动（react-window、tanstack-virtual）
- **预防方法**:
  - 在 SPEC 中定义分页策略
  - 性能测试时模拟大数据量场景
  - 代码审查时检查数据加载逻辑

### 12. 无 Loading 状态

- **症状**: 点击按钮或加载页面后，没有任何视觉反馈，用户不知道系统是否在处理
- **危害**:
  - 用户重复点击
  - 用户以为系统卡死
  - 体验差，显得产品不专业
- **检测方法**:
  - 测试所有异步操作：是否有 loading 指示
  - 检查按钮点击后是否有 `disabled` 或 `loading` 状态
  - 检查页面初始加载是否有骨架屏或 spinner
- **修复方案**:
  - 按钮：loading spinner + disabled
  - 页面：骨架屏（skeleton）或进度条
  - 数据加载：表格/列表行内 spinner
  - 全局：顶部进度条（NProgress、react-top-loading-bar）
- **预防方法**:
  - 封装 `AsyncButton` 组件，自动处理 loading
  - 在 `modes/run.md` 实现闭环中检查 loading 状态
  - 设计系统提供标准 loading 组件

### 13. 错误状态 = console.error

- **症状**: API 错误或异常只在控制台输出，用户界面没有任何提示
- **危害**:
  - 用户看到白屏或卡住，不知道发生了什么
  - 无法收集用户端的错误信息
  - 问题难以复现和定位
- **检测方法**:
  - 全局搜索 `console.error`、`console.log` 在异步代码中的使用
  - 断开网络测试：页面是否有错误提示
  - 检查 API 错误处理：是否有 `catch` 块，catch 后做了什么
- **修复方案**:
  - 统一错误处理：全局错误边界（Error Boundary）+ 错误提示组件
  - API 错误：根据状态码显示友好提示（401→请登录，500→服务器错误）
  - 记录错误日志到监控服务（Sentry、LogRocket）
  - 提供重试按钮或联系支持的方式
- **预防方法**:
  - 封装 `apiClient`，统一处理错误响应
  - 在 `modes/run.md` 实现闭环中检查"无控制台红色错误"
  - 测试用例覆盖网络错误场景

## API 反模式

### 14. API 路径写死

- **症状**: `fetch('/api/v1/users')` 直接写在组件中，不读配置文件
- **危害**:
  - 环境切换困难（dev/staging/prod）
  - 路径变更时需要全局替换
  - 无法做 API 版本管理
- **检测方法**:
  - 搜索 `fetch(`、`axios.`、`/api/` 在组件代码中的使用
  - 检查是否有统一的 API 配置文件
  - 检查 `.env` 中是否有 `API_BASE_URL`
- **修复方案**:
  - 提取 API 基础路径到环境变量或配置文件
  - 封装 `apiClient`，所有请求通过它发起
  - 使用 API 版本常量：`API_V1 = '/api/v1'`
- **预防方法**:
  - 在 `modes/run.md` 编码前门禁中检查 API 契约
  - 代码审查时禁止组件中直接写 API 路径
  - 使用 OpenAPI/Swagger 生成 API 客户端

### 15. 不处理 401/403

- **症状**: 用户未登录或权限不足时，页面白屏或显示错误数据
- **危害**:
  - 安全隐患：未授权用户看到敏感数据
  - 用户体验差：不知道需要登录
  - 无法区分"未登录"和"服务器错误"
- **检测方法**:
  - 测试未登录状态访问需要权限的页面
  - 检查 API 响应处理：是否有 401/403 分支
  - 检查路由守卫：是否有权限检查
- **修复方案**:
  - 401：跳转到登录页，登录后返回原页面
  - 403：显示"无权访问"页面，提供联系管理员的方式
  - 全局拦截器统一处理 401/403
  - 路由级别权限控制
- **预防方法**:
  - 在 SPEC 中定义权限模型
  - 封装 `useAuth` hook，统一处理认证状态
  - 测试用例覆盖未登录和权限不足场景

### 16. 无超时处理

- **症状**: API 请求没有设置超时，网络异常时请求卡住无限等待
- **危害**:
  - 页面一直 loading，用户无法操作
  - 资源泄漏，累积未完成的请求
  - 无法区分"慢网络"和"服务器无响应"
- **检测方法**:
  - 检查 `fetch` 或 axios 配置是否有 `timeout`
  - 模拟慢网络（Chrome DevTools Network Throttling）
  - 检查是否有请求取消逻辑（AbortController）
- **修复方案**:
  - 设置默认超时：10-30 秒
  - 使用 AbortController 支持请求取消
  - 超时后显示重试按钮
  - 长时间操作使用轮询或 WebSocket
- **预防方法**:
  - 在 `apiClient` 中统一配置超时
  - 在 `modes/run.md` 编码前门禁中检查超时策略
  - 性能测试包含慢网络场景

### 17. 重复请求

- **症状**: 快速点击按钮、搜索输入时触发大量重复请求
- **危害**:
  - 服务器压力增大
  - 竞态条件：后发出的请求先返回，显示错误数据
  - 资源浪费
- **检测方法**:
  - 快速点击按钮，检查网络面板请求数量
  - 快速输入搜索关键词，检查请求频率
  - 检查是否有防抖（debounce）或节流（throttle）
- **修复方案**:
  - 按钮：loading 状态禁用重复点击
  - 搜索：debounce 300-500ms
  - 自动保存：debounce + 取消上一个请求
  - 使用 React Query/SWR 的自动去重
- **预防方法**:
  - 封装 `useDebounce`、`useThrottle` hook
  - 在 `modes/run.md` 写文件前自检中检查防抖节流
  - 使用标准数据获取库（React Query、SWR、Apollo）

## 架构反模式

### 18. "use client" 滥用

- **症状**: 整个文件标记 `"use client"`，组件全部在客户端渲染
- **危害**:
  - 首屏加载慢，JS bundle 大
  - SEO 差，搜索引擎无法抓取内容
  - 失去 Next.js/App Router 的 SSR 优势
- **检测方法**:
  - 统计 `"use client"` 指令数量 vs 文件总数
  - 检查是否可以在组件级别拆分（子组件用 client，父组件 server）
  - 检查页面初始 HTML 中是否有内容
- **修复方案**:
  - 只在需要浏览器 API 的组件上使用 `"use client"`
  - 将表单、交互组件拆分为 client 子组件
  - 数据获取在 server component 中完成
  - 使用 Next.js 的 streaming 和 partial prerendering
- **预防方法**:
  - 在 `modes/run.md` 写文件前自检中检查 `"use client"` 必要性
  - 代码审查时质疑每个 `"use client"`
  - 使用 `@next/bundle-analyzer` 监控 bundle 大小

### 19. Import 路径混乱

- **症状**: 相对路径 `../../../components/Button` 层级过深，难以维护
- **危害**:
  - 文件移动后路径全部失效
  - 难以判断模块来源
  - 循环依赖难以发现
- **检测方法**:
  - 搜索 `../` 连续出现 3 次以上的 import
  - 检查 `tsconfig.json` 或 `jsconfig.json` 是否有 `paths` 配置
  - 检查是否有统一的别名规范
- **修复方案**:
  - 配置路径别名：`@/components`、`@/utils`、`@/types`
  - 使用 barrel exports（`index.ts`）简化导入
  - 按功能模块组织目录，而非按类型
- **预防方法**:
  - 在 `tsconfig.json` 中定义 `paths`
  - ESLint 规则限制相对路径层级
  - 代码审查时检查 import 路径

### 20. 类型定义和实现不同步

- **症状**: TypeScript 接口定义和实际 API 响应不一致，运行时报错
- **危害**:
  - 编译通过但运行失败
  - 类型安全形同虚设
  - 联调时才发现问题，修复成本高
- **检测方法**:
  - 检查是否有 `as any`、 `@ts-ignore`、`@ts-expect-error` 的滥用
  - 对比前端类型定义和后端 API 文档
  - 使用 `zod`、`yup`、`io-ts` 做运行时校验
- **修复方案**:
  - 使用 OpenAPI/Swagger 生成前后端共享类型
  - API 响应使用 schema 校验（zod）
  - 类型定义和 API 文档同步更新
  - 减少 `any` 使用，启用严格模式
- **预防方法**:
  - 在 `modes/run.md` 编码前门禁中检查类型策略
  - CI 中运行类型检查 `tsc --noEmit`
  - 代码审查时检查类型变更

### 21. 死代码不清理

- **症状**: 注释掉的代码、未使用的函数/组件/导入长期留在代码库中
- **危害**:
  - 代码库膨胀，维护困难
  - 新开发者困惑：这段代码还有用吗？
  - 构建产物包含无用代码
- **检测方法**:
  - 使用 `knip` 或 `depcheck` 检查未使用的导出和依赖
  - ESLint `unused-imports` 规则
  - 搜索大量注释掉的代码块
  - 检查 `package.json` 中是否有未使用的依赖
- **修复方案**:
  - 删除未使用的导入、变量、函数、组件
  - 注释掉的代码超过 2 周未恢复就删除（git 历史可找回）
  - 移除未使用的依赖
  - 使用 tree-shaking 友好的导入方式
- **预防方法**:
  - 在 `modes/run.md` 实现闭环中检查"无未使用导入"
  - CI 中运行 `knip` 检查
  - 代码审查时标记死代码
  - 定期（每迭代）清理死代码

### 22. 项目配置缺失（新增 — 来自测试沉淀）

- **症状**: 项目缺少 `vite.config.js`、ESLint 配置、`tsconfig.json` 等基础配置文件
- **危害**:
  - `npm run lint` 直接报错退出，代码风格无法统一
  - 无路径别名，import 使用 `../../../` 地狱
  - 构建行为不可控，无法自定义优化
  - 新开发者接手时无法快速理解项目规范
- **检测方法**:
  - 检查根目录是否有 `vite.config.*` / `webpack.config.*`
  - 检查是否有 `.eslintrc.*` / `eslint.config.*`
  - 检查是否有 `tsconfig.json` / `jsconfig.json`
  - 运行 `npm run lint` 验证是否通过
- **修复方案**:
  - 初始化时配置 ESLint：`npm init @eslint/config`
  - 配置路径别名：`tsconfig.json` 中定义 `paths`
  - 配置构建工具：vite.config 或 webpack.config
  - 将 lint/build 加入 CI 和 git hooks
- **预防方法**:
  - 在 `modes/run.md` 编码前门禁中强制"读取项目配置"
  - 项目模板自带完整配置文件
  - `/kit init` 时自动生成配置文件

### 23. 全部 inline style（新增 — 来自测试沉淀）

- **症状**: 所有样式使用 `style={{ ... }}` 内联写死，无 CSS 文件或 CSS-in-JS 方案
- **危害**:
  - 样式无法复用，每个组件重复写相同样式
  - 无法使用伪类（:hover, :focus, ::before）
  - 无法使用媒体查询做响应式
  - 暗色模式适配几乎不可能
  - 构建产物大（每个 style 对象都是 JS）
- **检测方法**:
  - 统计 `style={{` 在代码中的使用频率
  - 检查 `src/` 下是否有 `.css`、`.scss`、`.less` 文件
  - 检查是否有 Tailwind/styled-components/Emotion 等方案
- **修复方案**:
  - 引入 Tailwind CSS（推荐）：`npm install -D tailwindcss postcss autoprefixer`
  - 或使用 CSS Modules：`import styles from './Button.module.css'`
  - 将 inline style 迁移到 className
  - 建立设计 token 系统
- **预防方法**:
  - 在 `modes/run.md` 编码前门禁中"声明 UI 工具链"
  - 项目初始化时即配置 CSS 方案
  - 代码审查时拒绝纯 inline style 的组件

### 24. 加载逻辑闪烁（新增 — 来自测试沉淀）

- **症状**: `loading` 初始值为 `false`，`useEffect` 中才设为 `true`，导致先显示数据再显示 loading
- **危害**:
  - 用户看到内容"闪一下"然后变成 loading
  - 显得产品不专业，像有 bug
  - 用户体验差，可能误以为数据丢失
- **检测方法**:
  - 检查 loading 状态的初始值（应为 `true`）
  - 用慢网络（Fast 3G）模拟，观察是否有闪烁
  - 检查 `useEffect` 中 loading 的 set 顺序
- **修复方案**:
  - `const [loading, setLoading] = useState(true)`
  - 骨架屏和真实内容互斥显示
  - 数据获取完成后再关闭 loading
- **预防方法**:
  - 封装 `useAsync` hook，统一处理 loading 状态
  - 在 `modes/run.md` 实现闭环中检查 loading 状态逻辑
  - 测试用例覆盖慢网络场景
